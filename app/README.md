# Blink tester app (`app/`)

This directory holds the **production deployment channel** for Blink copy-paste:
a signed, notarized Swift `.app` that ships to non-developer testers as a `.dmg`.
It is a second, independent loop from the local research harness (`scratchpad/`) —
no runtime dependency either way.

> Heads up: this directory is scaffolding. Some phases are implemented and testable
> (Python runtime, artifact bundles, import script); others (Xcode project, DMG
> packaging, notarization) are wired up but not yet exercised end-to-end. Track
> phase status in `docs/EXPERIMENT_LOG.md`.

See:

- `../README.md` — repo overview; points at both loops.
- `../CLAUDE.md` — repo implementation guide.
- `../docs/ARTIFACT_SCHEMA.md` — v1 bundle contract shared by both loops.
- `../docs/PROJECT_BRIEF.md` — product scope.

## Two loops, one contract

```
Research loop (unchanged)              Tester deployment loop
━━━━━━━━━━━━━━━━━━━━━━━━━━━            ━━━━━━━━━━━━━━━━━━━━━━━━━━
./capture (dev venv)                   Blink.app (bundled Python)
  └─ scratchpad/run_gemini_trial.py      ├─ Swift: hotkeys, capture,
  └─ writes scratchpad/fixtures/         │         AX metadata, insert
                                         └─ app/python/run_once.py
./sweep                                     └─ writes
  └─ reads scratchpad/fixtures/              ~/Library/Application Support/
  └─ reads scratchpad/field_runs/  ◄───────  Blink/runs/<ts>/
                                                    │
                                          scratchpad/import_field_runs.py
                                                    │
                                          ingests → scratchpad/field_runs/
```

The only cross-loop coupling is the **versioned bundle schema**
(`docs/ARTIFACT_SCHEMA.md`). Both emitters produce v1 bundles; `./sweep` reads
both.

  scripts/

    build.sh                      xcodebuild + copy python-dist

No symlinks between `app/python/` and `scratchpad/`. The fork's header comment
records the source SHA. Re-syncing is a deliberate commit.

## Production Python (Phase 1, usable today)

`app/python/run_once.py` is a clean CLI, independent of `scratchpad/`:

```bash
python3 app/python/run_once.py \
  --source /path/to/source.png \
  --target /path/to/target.png \
  --target-meta /path/to/target_metadata.json \
  --caret /path/to/caret.json \
  --settings /path/to/settings.json \
  --prompt /path/to/prompt.txt \
  --out-dir /tmp/blink-runs
```

Writes `<out-dir>/<ts>/` containing `fixture.json`, `source.png`, `target.png`,
`run.json`, `output.txt`, `target_metadata.json`, `settings.json`, and
`caret.json` when the caller captured AX selected-range state. Sweep reads the
bundle via `./sweep --fixtures '/tmp/blink-runs/*' --configs …`.

`output.txt` preserves the raw model output for sweep parity. stdout carries the
paste-normalized text (for Swift to capture and paste). stderr carries progress
logs. `--skip-gemini` emits the bundle without a live API call - useful for
bundle-shape verification.

Requires `GEMINI_API_KEY` in the environment. For proxy/bearer auth (Phase 5),
also set `BLINK_PROXY_URL` and optionally `BLINK_PROXY_TOKEN`; both are
forwarded into `http_options` when building the client.

## Capability token, not auth

`Info.plist.BlinkProxyToken` is a **hardcoded shared bearer token** — a public
capability, not authentication. It's fine for dogfood but do not treat leaks of
the notarized `.app` as equivalent to user credentials. Real auth lands after
field results justify it.

## Building the app

Prerequisites:

- macOS 14+ with Xcode command-line tools
- `xcodegen` (`brew install xcodegen`)
- `create-dmg` for packaging (`brew install create-dmg`)
- Apple Developer account: `TEAM_ID` and `BUNDLE_ID` in `app/scripts/config.env`
  (see `config.env.example`)

End-to-end build:

```bash
bash app/scripts/fetch_python.sh    # one-time: downloads python-build-standalone
xcodegen generate --spec app/project.yml
bash app/scripts/build.sh           # xcodebuild → Blink.app with bundled python
bash app/scripts/sign.sh            # leaf-first codesign with hardened runtime
bash app/scripts/notarize.sh        # notarytool submit --wait + stapler staple
bash app/scripts/make_dmg.sh        # → build/Blink-<version>.dmg
```

Each step is idempotent and safe to re-run.

For local profiling / dogfood, prefer the canonical installer instead of
opening `DerivedData` or `app/build` products directly:

```bash
bash app/scripts/install_local_app.sh
```

That script:

1. fetches `python-dist` if needed,
2. builds a self-contained Release app,
3. installs it to `~/Applications/Blink.app`,
4. moves duplicate Blink app bundles into `.context/disabled-apps/*.app.disabled`,
5. optionally resets TCC with `--reset-tcc`, and
6. relaunches the canonical app.

The goal is to keep Spotlight, LaunchServices, and TCC pointed at one stable
local Blink install.

For a full dogfood session — clean build, clean permissions, and guaranteed
capture of fixtures / profiling timings / debug logs per trial — follow
[`../docs/DOGFOOD_PLAYBOOK.md`](../docs/DOGFOOD_PLAYBOOK.md).

## Permissions

First-launch `PermissionsWindow` walks the tester through three grants:

1. **Input Monitoring** — required for `CGEventTap` to see global hotkeys.
2. **Accessibility** — required for AX metadata capture and Cmd+V insertion.
3. **Screen Recording** — required for `ScreenCaptureKit` window/region capture.

All three are documented in `scratchpad/README.md:40-44` for the research-loop
equivalent.
